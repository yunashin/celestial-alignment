# CLAUDE.md — working notes for this codebase

Context for whoever (human or agent) picks up this repo next. Read this before making changes — several things here are non-obvious from the code alone.

## Internationalization (i18n)

Every user-facing string in the app — static UI chrome, sign/element/surge/power-up/damage-card flavor text, and every dynamic gameplay log message the engine generates — lives in YAML, not inline in TSX/TS. Two locales exist today (`en`, `ko`); `ko.yaml` is currently a byte-for-byte copy of `en.yaml`'s English text (a deliberate placeholder — the user fills in real Korean translations later), but the key SET in both files must always match exactly, enforced by a dev-time console warning (see below).

- **`src/i18n/en.yaml` / `src/i18n/ko.yaml`** — nested YAML dicts, dot-path addressable (e.g. `signs.ARIES.label`, `log.moveMulti`). Imported as raw text via Vite's `?raw` suffix (`import enRaw from "./en.yaml?raw"`) and parsed with `js-yaml`'s `load()` at module init — this needed a new `src/vite-env.d.ts` (`/// <reference types="vite/client" />`) for `?raw` imports to typecheck, since the project had no ambient Vite types file before. Adding a 3rd language is: create `xx.yaml` with the same key structure as `en.yaml`, add `{ code: "xx", nativeLabel: "..." }` to `LANGUAGES` in `i18n/index.ts`, add it to the `RESOURCES` map — nothing else needs to change.
- **`src/i18n/index.ts`** — the core engine. `t(key, params?)` is a plain, non-reactive module-level function (reads a module-level `currentLocale` variable) usable from pure engine code with zero React imports — this is what makes it possible for `reducer.ts`/`eclipse.ts`/`rules.ts` to build translated log messages without becoming React-aware. `useTranslation()` wraps the same `t` in `useSyncExternalStore` so components re-render on locale change; both share one translation surface, there's no parallel/duplicated lookup logic. Interpolation is `{paramName}` tokens replaced via regex from a `Record<string, string | number>`. **Fallback chain: current locale → English → the literal key string itself.** That last fallback is why several engine unit tests that pass arbitrary literal English strings as a fake "key" (e.g. a hardcoded `damageMessage: "Test damage message."` in test setup) still pass unmodified — `t()` returns an unrecognized key verbatim, so a raw string used AS a key just round-trips through untouched. Locale persists to `localStorage` under `celestial-alignment:locale` (same `celestial-alignment:` prefix convention as `utils/setupStorage.ts`). A dev-time IIFE at the bottom of `index.ts` recursively diffs every locale's leaf-key set against English's and `console.warn`s on any mismatch (missing or extra keys) — runs unconditionally, not gated behind `import.meta.env.DEV`, since the check is cheap and catching a drifted key immediately beats silently falling back to English for it.
- **`src/i18n/gameText.ts`** — small typed helpers (`signLabel`, `signAbility`, `signDesc`, `elementLabel`, `elementDescription`, `surgeText`, `powerUpText`) that take a `TFunc` (either hook's `t` or the plain engine `tr`) as their first argument, so call sites read as "give me this sign's label" instead of hand-assembling `` `signs.${sign}.label` `` inline everywhere. `powerUpText` additionally folds numeric tuning constants (`SHOOTING_STAR_TRACKER_DOWN_PCT`, `SHOOTING_STAR_AP_BONUS`, `SHOOTING_STAR_HAND_BONUS`, `SHOOTING_STAR_SELF_HEAL_AMOUNT`, `STARTING_HP`) into the `t()` call as interpolation params — this keeps `constants.ts` the single source of truth for those numbers even though the surrounding sentence now lives in YAML, rather than duplicating the numbers into the YAML text itself.
- **`reducer.ts` and `eclipse.ts` import the translate function under the alias `tr`** (`import { t as tr } from "../i18n"`), not plain `t` — both files use `t` extensively as a local `Tile` variable name (`for (const t of row)`, `const t = s.tiles[x][y]`), so importing it as plain `t` would silently shadow every one of those. `rules.ts` has no such collision and imports plain `t`. If you add translation calls to a new engine file, check for this collision before picking an import name.
- **`constants.ts`'s `SIGNS`/`ELEMENT_META` now hold ONLY structural/visual fields** (glyph, element, impl status, color, soft) — every translatable label/ability/desc/description moved to `signs.<SIGN>.*`/`elements.<ELEMENT>.*` in YAML, read via `gameText.ts`'s helpers. `SURGE_META`/`POWER_UP_META` were removed entirely (replaced by `surge.<ELEMENT>`/`powerUps.<POWERUP>` YAML keys + `surgeText`/`powerUpText`). `DAMAGE_CARDS`' `DamageCardMeta.messageKey` holds a YAML key (e.g. `"damageCards.fire"`), not resolved text — `EclipseCard.damageMessage` (set from `messageKey` in `buildEclipseDeck`) stores that same key, and it's only resolved to actual text via `tr()` inside `resolveEclipse` at the moment the card actually fires, not at deck-build time — so a Damage Card built under one locale still displays correctly if the locale is switched before it resolves.
- **Dynamic gameplay log messages "freeze" at generation time.** Every `log()`/`important()` call in `reducer.ts`/`eclipse.ts` resolves its `tr("log.xxx", {...})` call immediately and bakes the resulting plain string into `GameState.log`/`messageLog` — switching languages mid-game does NOT retroactively re-translate history already on screen, the same way most apps don't retranslate old toast notifications after a language switch. Conditionally-appended sentence fragments (e.g. Cosmic Draw's balanced-hand suffix, a chain event's "eases by X%" vs. "no further effect" tail) are separate YAML keys concatenated in JS exactly mirroring the original ternary-based string-building, specifically so English output stayed byte-identical to the pre-i18n text (this made cross-checking against the pre-existing test suite's substring assertions straightforward — most needed zero changes).
- **`src/components/LanguageSwitcher.tsx`** is a small EN/한국어 toggle (`useTranslation()`'s `locale`/`setLocale`), mounted in both `SetupScreen` and `GameScreen`'s header. In `GameScreen`, the header row is a `grid-cols-[1fr_auto_1fr]` layout (Back button / centered title / right-aligned LanguageSwitcher+SeedDisplay cluster) rather than the earlier centered-flex-plus-absolutely-positioned-siblings layout — the absolute-positioning approach let the right cluster visually overlap the centered title once the LanguageSwitcher widened it, since absolutely-positioned siblings are pulled out of flow and can't push against a `justify-center`'d sibling. The grid version gives each cluster its own track, so it can never intrude into the title's space regardless of viewport width; the right cluster also wraps (`flex-wrap`) onto its own line if it still doesn't fit.
- **Exception: buttons whose label shows an underlined literal English letter as its keyboard shortcut stay hardcoded English in JSX, never translated** — e.g. `<u>M</u>ove`, `<u>P</u>urify`, `Discard to <u>H</u>eal`, `<u>C</u>osmic Draw`/`<u>C</u>onfirm`, `<u>E</u>nd Turn`/`H<u>e</u>al`, `◂ <u>B</u>ack`, `Protecti<u>v</u>e Precision`, `<u>T</u>erraform Hand`, `<u>R</u>otate ↻`. The underline marks a specific English letter that IS the keyboard shortcut (see the "Keyboard shortcuts" bullet under UI conventions below) — translating the word would either break the visual correspondence between the underlined letter and the actual key, or require a per-locale shortcut remap that doesn't exist. `ControlPanel.tsx` is the main place these live; check there before assuming a button's text should move to YAML.
- **`npx tsc --noEmit` and `npx vitest run` both need to stay green after any i18n change** — the YAML files are typed as `unknown`/`Dict` (js-yaml's `load()` return type), so a malformed key path or a genuinely missing key won't be caught by TypeScript, only by the runtime key-parity `console.warn` (dev-time) or a test that asserts specific translated text.

## Gameplay nuances that aren't obvious from a skim

- **Purify is path-based, not network-based.** A player can only purify a tile they've personally walked onto via `MOVE` (tracked in `Player.visited`), not any tile in their element's connected network. `visited` is seeded with the player's spawn tile and grown on every successful `MOVE`. Placing a card at a tile does **not** add it to `visited` — the pawn didn't walk there.
- **`Player.visited` is `Record<string, true>`, not a `Set`.** This is deliberate: `gameReducer`'s clone step is `JSON.parse(JSON.stringify(...))`, which silently turns a `Set` into `{}`. Any new per-player "have they touched X" state must use a plain object/array, not a `Set`, or it will evaporate on the next dispatch.
- **`computeNetwork(tiles, element)` traces connectivity through *any* card on the board, regardless of which element originally placed it.** This is what makes Corruption/Surge scaling and win-glow convergence meaningful — two different elements' paths can legitimately share physical tiles near the center. Don't "fix" this to filter by `card.element`; it's load-bearing. Its optional 5th parameter, `crossCenter` (default `true`), controls whether the traversal keeps exploring back out the Orrery's OTHER sides once it reaches the center — `isPathComplete` passes `false` specifically so its purity scan doesn't tunnel through the Orrery into an unrelated element's branch (see that bullet below).
- **Placement validity already implies connectivity.** This is why shooting stars don't need a separate "is this connected" check: `validPlacement` (in `rules.ts`) rejects any placement that doesn't link back to a live network/node/center, so the moment a card successfully lands on a shooting-star tile, `activateShootingStar` fires unconditionally in `reducer.ts`'s `PLACE` case.
- **`apBonus` / `handSizeBonus` are permanent, global, additive state** granted by the AP/hand-size shooting-star power-ups. `handSize(s, p)` and the AP reset in `END_TURN` both read from `s`, not just `p` — if you add a new per-player hand-size or AP modifier, thread it through the same two call sites (`initGame`, `END_TURN`, `DISCARD`) or it'll be silently ignored on refill/reset.
- **`lastActedPlayerId`** is set unconditionally at the top of `gameReducer` right after cloning (`s.lastActedPlayerId = p.id`), *before* the switch statement's per-action validation. This is safe because invalid actions `return state` (the original, pre-clone object), discarding `s` entirely — so the field only ever "sticks" on actions that actually succeed. Keep this ordering if you refactor the reducer.
- **`Tile.isShielded`/`shieldOwner` is Virgo's Protective Precision** (a 2x2 area, once per turn, protects from both Eclipse effects AND the traveling asteroid, cooldown via `GameState.virgoShieldCooldown`) — it used to be Capricorn's single-tile "Sovereign Peak" before that ability was replaced with Terraform (`CONVERT_HAND_EARTH`, converts the whole hand to Earth cards for 1 AP, repeatable), and was previously called "Bastion Ward" before it gained asteroid protection alongside the rename. Don't confuse `Tile.isShielded` with `eclipse.ts`'s `isShielded(s, target)` helper — that one is Cancer's Lunar Shield damage check on *players*, a completely separate mechanic that happens to share a name. Eclipse Corruption/Void simply skip `isShielded` tiles when picking targets; there's no separate damage-immunity check needed because the tile can never be seized in the first place. `triggerAsteroidShift` (see below) excludes `isShielded` tiles from both its destination-candidate pool and its per-tile sweep destruction, the same way it already treats completed-path/Cancer-shield/enclosed-loop tiles — but note this is CARD/tile protection only, not player-damage immunity: a Guardian standing on a Virgo-shielded tile still takes the asteroid's 1 damage exactly like anyone else (only Cancer's Lunar Shield, via `damage()`'s own Cancer check, blocks player damage).
- **`GameState.virgoShieldCooldown` is set to `2` on use, not `1`, despite reading as a "1-turn cooldown" to players.** It only decrements at the moment control returns to the Virgo player specifically (`END_TURN`'s `if (newActive.sign === "VIRGO" && s.virgoShieldCooldown > 0) s.virgoShieldCooldown -= 1`, unaffected by however many other players' turns pass in between). Starting at `2` means: disabled for the rest of the turn it was cast (cooldown untouched, still >0), the first decrement (arriving back at Virgo's very next turn) leaves it at `1` — still disabled that turn — and the second decrement (the turn after that) clears it to `0`. A starting value of `1` would incorrectly clear the cooldown on Virgo's very next turn instead of the turn after. If you touch this, `reducer.test.ts`'s "Virgo's Protective Precision" describe block walks the full turn sequence and will catch a regression.
- **Scorpio's Regeneration is a manual, targeted heal** (`SCORPIO_HEAL`: discard a hand card, once per turn, to heal *any* Guardian 1 HP — including themselves), not an automatic Eclipse-nullify. Pisces heal is capped at 2 HP per discard action, not uncapped per card.
- **Players spawn standing on their own edge node tile** (`initGame` sets `position` to `s.nodes[element]` directly), not on an adjacent board tile. A node has up to 3 in-bounds neighbors (the 4th direction always runs off the board), and a card can anchor to a node from *any* of those sides — `computeNetwork`/`elementPathDepths` in `board.ts` both loop over all of a node's in-bounds sides (mirroring how `enterCenter()` already tries all 4 sides of the Orrery) rather than only the single `nodes[element].dir` lane. `validPlacement`'s `check()` reflects this too (`if (nt.node) return true`, no direction match required). Don't reintroduce a single-direction-only node entry — that was an earlier bug where only the Orrery-facing tile was usable and the other two adjacent tiles were dead ends.
- **The traveling asteroid (`triggerAsteroidShift`)** fires on a hardcoded cadence per player count (`ASTEROID_SHIFT_INTERVAL` in constants.ts: 2p=4 turns, 3p=5, 4p=6 — previously a random 4-6 turn window regardless of player count; `asteroidShiftInterval()` in reducer.ts looks it up, used both at `initGame` and every time `GameState.turnsUntilAsteroidShift` resets after firing), relocating a random existing asteroid to a random tile ≥3 tiles away, in a straight line (`lineTiles`, Bresenham). Every tile it crosses — including the destination — has its card/corruption/lock/shield/shooting-star state wiped and deals 1 damage to any player standing there (via the same `damage()` used by Eclipse, so Cancer's shield still applies), **except tiles that are part of an already-completed element path, a Cancer Lunar Shield radius, an enclosed loop, a Virgo Protective Precision shield, or a tile a Guardian is currently standing on** (completed-path tiles computed via `isPathComplete`/`computeNetwork(..., false)` against the current player elements before picking a destination or clearing tiles — see `completedPathTiles` in `triggerAsteroidShift`, and the crossCenter bullet below for why that trailing `false` matters; `Tile.isShielded` is checked directly, no computed set needed; the occupied-tile check reuses the same `occupied` set already built for destination selection, but additionally applied per-tile along the whole path, not just the destination — a Guardian standing on the swept tile still takes the 1 damage, only their card is spared, specifically so the asteroid can't both hurt them AND strand them on bare ground). The asteroid's DESTINATION candidates also exclude `isVoid` tiles — an asteroid can never come to rest on an existing Black Hole, mirroring how Eclipse Void's own target filter already excludes `isAsteroid` tiles in the other direction. `Tile.asteroidHitStep` records order along the path purely for staggering the sweep animation; it's never cleared, which is fine because the one-shot CSS animation only plays once when the value first changes (see the shooting-star `powerUpFlash` pattern below). A tile whose card is ACTUALLY destroyed (not just protected/spared) also gets `Tile.explosionStep = i` (same per-tile path index as `asteroidHitStep`), driving a separate, more dramatic burst-plus-debris animation in `TileView` layered on top of the ordinary sweep flash — this is the signal that something was really destroyed here, not just grazed. Each tile the asteroid crosses gets its own log line (card destroyed / path preserved / shield deflected / Guardian held their ground), not just a single summary.
- **Building the asteroid's (and Eclipse Corruption's) "protected tiles" set from a completed path must call `computeNetwork(..., crossCenter: false)`, not the default `true`.** Both `triggerAsteroidShift` (in reducer.ts) and Corruption's `sealed` set (in eclipse.ts) walk `isPathComplete`-confirmed elements' networks to build a set of tiles that should be immune. The default `crossCenter: true` traversal, upon reaching the Orrery, keeps exploring back out its OTHER sides — which is exactly what you want for connectivity/win-glow purposes (see `computeNetwork`'s own doc comment) but is wrong here: it tunnels from one element's genuinely complete, pure network through the Orrery hub into a completely different element's network, and if that OTHER element's path happens to be incomplete or impure, its tiles get wrongly swept into the "protected" set anyway, just because they're also attached to the center. A short, two-card fragment sitting on the Orrery with no real path back to its own node should stay fully vulnerable to both Corruption and the asteroid — passing `false` (matching `isPathComplete`'s own internal usage) keeps the traversal from ever leaving the confirmed-complete element's own network in the first place.
- **Grouped log messages must be built up and flushed with `logGroup`, not several direct `log()` calls.** `log()` unshifts (newest at index 0), so calling `log()` in chronological order for a "header + per-tile follow-up" event (like the asteroid sweep) makes the follow-ups render *above* their header, in reverse order — visually disconnected from the arrow (`↳`) that's supposed to tie them to it. `logGroup(s, messages)` takes messages in narrative (top-to-bottom) order and logs them back-to-front internally so the whole group reads correctly once flushed. `damage()` takes an optional `out?: string[]` for exactly this reason — pass it through when a caller (e.g. `triggerAsteroidShift`) needs the damage line to land in the right spot within its own group instead of jumping to the top of `s.log` on its own.
- **Leo's Solar Flare traces a live connector path, not adjacency.** `PURIFY`'s Leo branch calls `tracePathBetween(tiles, player.position, target)` (BFS over Star Card connectors, treating any node/the Orrery as an always-open hub) and cleanses every corrupted tile on that route, not just the 4 tiles physically touching the purified one. Unlike `computeNetwork`, `tracePathBetween` deliberately ignores corruption while routing — otherwise the beam could never punch through a corrupted stretch to reach whatever corrupted card lies beyond it. If no connector route exists between the player and the target, only the target tile itself gets purified (the normal, non-Leo behavior).
- **A 3+ SAME-ELEMENT card chain gives an Eclipse Tracker discount that keeps growing every time the chain is extended further — no cross-element "wildcard" counting anymore.** `computeSameElementChainGroup(tiles, element, x, y)` (in `board.ts`) flood-fills the physically-connected blob of cards sharing exactly one `element`, starting from (and requiring) a card of that element at `(x,y)` itself — a foreign-element card bridged onto the end of a chain simply isn't part of it, full stop, no eligibility/node-attachment check needed at all (that whole mechanism, along with `computeChainGroup`/`computeEligibleChainGroup`/`Tile.wildcardFlash`/`caWildcardFlash`, was deleted along with the wildcard concept). Like `computeNetwork`, this tolerates corruption — a corrupted card in the middle of an otherwise-continuous same-element chain doesn't sever it, since the card's connector shape is still physically there even while seized. The discount applies on **every** placement that leaves the group at `chainAfter >= CHAIN_LENGTH_THRESHOLD` (3) — not just the one that first crosses it — and scales with how far past the threshold the chain now sits: `reduction = (CHAIN_TRACKER_DISCOUNT_OWN if own element else CHAIN_TRACKER_DISCOUNT) + CHAIN_TRACKER_BONUS_DISCOUNT * chainExtraLength`, where `chainExtraLength = chainAfter - CHAIN_LENGTH_THRESHOLD + 1` (so reaching exactly 3 gives a `chainExtraLength` of 1, reaching 4 gives 2, and so on) — extending an already-3+ chain further keeps discounting the Tracker at a bigger rate each time, not just re-firing the visual event with no further effect. A single placement that both extends a long-standing chain AND completes a path (see the 4-player `PATH_COMPLETE_TRACKER_REDUCTION_4P` bullet below) legitimately gets both reductions at once — this isn't a bug, just two independent discounts stacking on the same dispatch. The log/message glyph (`{glyph}` in `log.chain`) is repeated once per `chainExtraLength`, capped at 5 repeats, purely so a very long chain doesn't print a wall of emoji — the printed card count already conveys the actual size. `lastChainEvent` is `{ tiles, start, end }` — `start` is whichever chain tile sits closest, Manhattan-wise, to the chain's own element's node; `end` is simply the tile that was just placed — recomputed fresh on every qualifying placement, not just the first. `GameScreen`'s big seq-diffing effect reacts to this the same way it reacts to other seq events: on change, it sets a local `chainGlowTiles` state to every tile in the group and clears it via `setTimeout` after 3000ms, rendered in `TileView` as a `caChainGlow` pulsing border/box-shadow for that entire window — a genuinely temporary highlight that fully reverts, and (being seq-driven rather than a per-tile latch) can fire again for an entirely separate later chain event even on tiles that already glowed once before.
- **A corrupted card decays and is destroyed outright after `CORRUPTION_DECAY_TURNS` (3) of its own placer's turns, unless purified first.** `Tile.placedBy` (player id) is stamped on every `PLACE`, unconditionally, regardless of whether the card ever gets corrupted — cheap bookkeeping that pays off only if corruption later strikes. `eclipse.ts`'s CORRUPTION branch sets `Tile.corruptionTurnsLeft = CORRUPTION_DECAY_TURNS` the moment it seizes a card. **The decay/destroy sweep runs at the very START of `reducer.ts`'s `END_TURN` case — before `resolveEclipse` (which is what might corrupt something new THIS dispatch) and before the active-player reassignment — keyed to `p` (the player who is ENDING their turn, captured pre-reassignment), not `newActive` (the player about to start the next one).** This ordering matters: keying it to `newActive` (an earlier version of this feature did) destroys the card during the transition INTO the placer's next turn — before they ever get to act on it — so a player watching the countdown hit 1 would see the card simply vanish before their following turn even started, with no turn where it read 0. Keying it to `p` instead means the decrement (and, on the final one, destruction) happens at the END of the placer's own turn — i.e. right after their 3rd turn since corruption, if they still haven't purified it by then, exactly matching the countdown they saw during that turn. No off-by-one adjustment is needed on the starting value (a literal 3, decrementing straightforwardly) — Corruption striking a card is never something that needs discounting against the same dispatch it happened in, since the sweep runs BEFORE `resolveEclipse` can freshly corrupt anything that turn. When it hits 0, the card is destroyed immediately (same `t.card = null` + flag-reset shape as asteroid destruction, pushed to `s.starDiscard` so it can reshuffle back into the Star Deck) and logged/`important()`-marked. Purifying a tile (`PURIFY`, including every tile Leo's Solar Flare sweeps) clears `corruptionTurnsLeft` back to `null` — a purified card is permanently safe (see the `isPurified` bullet elsewhere), full stop, decay isn't still ticking in the background. Asteroid destruction's existing full-tile-wipe block also resets both `placedBy`/`corruptionTurnsLeft` to `null`, mirroring how it already resets `isPurified`. **The number shown centered on the tile is a TOTAL turn count across every Guardian, not just the placer's own** — `rules.ts`'s `totalTurnsUntilCorruptionDecay(s, placedBy, corruptionTurnsLeft)` simulates END_TURN's own active-player rotation (including skipping Stasis players, capped at a generous iteration count and returning `null` if the placer is stuck in Stasis indefinitely) forward from `s.active`, counting every player's turn — including whichever turn is currently in progress — until the placer's own turns exhaust `corruptionTurnsLeft`. `GridBoard` computes this per corrupted tile and passes it to `TileView` as `corruptionTotalTurns`; `TileView` displays it (falling back to the raw placer-only `corruptionTurnsLeft`, then `"?"`, if the total can't be resolved) in the exact centered spot the old static `✖` mark used to occupy, and the tooltip spells out BOTH numbers explicitly ("N more turns total... — {placer} has M of their own turns left"). **If the placer has never personally visited the corrupted tile** (`Player.visited` — see the path-based-Purify bullet above; placing a card at a tile does NOT count as visiting it), the tooltip appends a plain-language note that they specifically can't purify it themselves ("{placer} can't purify this card since they haven't visited it yet.") — a real, common case, since Sagittarius (and any sign, before their pawn ever walks past its own spawn) can easily place a card at a tile its own placer has never set foot on. **Destruction by decay also gets its own distinct animation**, separate from the asteroid's explosion: `Tile.crumbleStep` (set to `s.turn` the instant the decay sweep destroys a card, `null` otherwise) drives a purple-tinted "dissolving to dust" flash plus downward-drifting debris (`caCrumbleFlash`/`caCrumbleDust` in styles.ts) — the same one-shot-animation-on-value-change trick as `asteroidHitStep`/`explosionStep` (`s.turn` only ever increases, so a later crumble on the same tile always gets a fresh value and replays), deliberately styled differently (purple/downward vs. orange/outward-bursting) so the two destruction causes read as visually distinct events.
- **The board is rectangular (`WIDTH`×`HEIGHT` = 19×11) with a randomized Orrery and node layout, generated fresh every game.** `Tile.x` is the row index (0..HEIGHT-1), `Tile.y` is the column index (0..WIDTH-1) — this predates the rectangular board and is easy to get backwards. `board.ts`'s `pickCenterAndNodes()` places the Orrery somewhere in the center `CENTER_ZONE_WIDTH`×`CENTER_ZONE_HEIGHT` tiles, then picks a position on each of the 4 edges (per `NODE_EDGE`) that's the same Manhattan distance `D` from the Orrery. Because the board is much wider than it is tall, that distance isn't always simultaneously achievable on the short (row) axis for an off-center Orrery — `pickAlong()` clamps the lateral offset to whatever fits in bounds, so equidistance is exact in the common case and only off by a tile or two in rare, very-off-center cases. `GameState.center`/`GameState.nodes` carry the chosen layout for the rest of the engine (nothing hardcodes positions anymore); `GridBoard`'s edge labels track the actual node position along their edge via a percentage, not a fixed centered spot.
- **Virgo's Protective Precision is a two-click preview-then-confirm flow**, not instant-on-click. `GameScreen` tracks `shieldPreview: {x,y} | null`; the first click on a valid anchor sets the preview (rendered via `GridBoard`'s `previewTiles` prop → `TileView`'s `previewed` styling, solid gold, distinct from the dashed cyan "clickable" highlight), and clicking the *same* anchor again dispatches `VIRGO_SHIELD`. Clicking a *different* valid anchor just moves the preview. This is the pattern to copy for any future "commit to an area, not a single tile" action.
- **A tapped hand card also gets a lightweight ghost preview on whichever highlighted tile currently has "focus"**, computed in `GameScreen` (`cardPreview`: the armed card's connections — rotated if Aquarius has set a manual `rotation` — plus its element color) and threaded through `GridBoard` (which only passes it to tiles that are actually `highlighted`, i.e. legal placement targets) down to `TileView`. This is deliberately NOT the same thing as Virgo's gold `previewed` styling above — it's a low-opacity, dashed-outline `CardGlyph` that costs nothing to dismiss and confirms nothing on its own; "focus" here means either the keyboard board cursor (`cursorFocused`, opacity forced to 0.7) or the tile being under the mouse (via the same `group-hover/tile` mechanism `TileView` already uses elsewhere for scoping hover-only effects, opacity 0.6) — both read off the identical `cardPreview` prop, so there's no separate hover-vs-keyboard code path to keep in sync.
- **`ControlPanel` is a two-tab layout**: a "Guardian" tab (default) with the Eclipse Tracker, active player's status card, action buttons, and roster — all with a little more breathing room and slightly larger text than before, since the Event Log no longer has to share the same scroll region — and a separate "Event Log" tab (labeled with the current turn number) holding the full, unfiltered `state.log`. `tab: "status" | "log"` is local `useState` in `ControlPanel`, defaulting to `"status"`; switching tabs is purely a rendering choice; it doesn't affect `mode`/`selectedCard`/any dispatch-relevant state elsewhere in `GameScreen`; nothing needs to be forced back to the status tab automatically on a new turn.
- **Aquarius can manually rotate a selected card before placing** via a `rotation: number | null` state in `GameScreen` (0-3 quarter-turns) and a "Rotate" button/keyboard shortcut (`R`) shown only when `mode === "place" && sign === "AQUARIUS"`. `getValidPlacements`/`validPlacement` take an optional `forcedRotation` that, when set, checks *only* that exact orientation instead of auto-trying all 4 — the chosen rotation is threaded through to the `PLACE` action itself (`{ type: "PLACE", ..., rotation }`) so the reducer places the same orientation that was highlighted, rather than re-deriving it and potentially picking a different valid rotation. `rotation` resets to `null` (back to auto-try-all) whenever a new card is armed.
- **End Turn asks for confirmation** when `hasAnyAction(state)` is still true at the moment it's clicked (`GameScreen`'s `requestEndTurn` → `EndTurnConfirmModal`), so the same "nothing left" check drives both the button's urgent glow *and* whether a confirmation is needed — if there's genuinely nothing else to do, End Turn fires immediately with no modal.
- **Connectivity tolerates corruption; *winning* doesn't.** `computeNetwork`/`computeCenterNetwork` no longer stop at a corrupted card — you can place new cards that connect through/past corrupted ground, so a partially-corrupted network keeps growing. `isPathComplete` is where purity is actually enforced: after confirming the network reaches the Orrery, it separately walks every tile in that network and fails if *any* of them is corrupted. This means Eclipse Corruption on an otherwise-finished path doesn't just cosmetically mark a tile — it genuinely un-wins that element until it's purified again. `isPathComplete` calls `computeNetwork(..., false)` (`crossCenter: false`) for this, deliberately NOT the default center-crossing traversal — otherwise one element's own genuinely complete, pure path could be wrongly reported incomplete just because a totally different element's corrupted card happens to share a "network" with it only by tunneling through the Orrery hub. The tile that actually proves "reaches the Orrery" is always found via the normal outward walk from this element's own node regardless of `crossCenter`, so restricting the traversal here never removes anything needed to prove the element's own path is complete — only tiles that were only reachable by crossing *through* center into someone else's branch.
- **`GameScreen`'s `litKeys` memo (the pulsing-glow "this path is complete" board highlight) must call `computeNetwork(..., false)` too, for the identical `crossCenter` reason as `isPathComplete` above — this was a real bug, not just a theoretical risk.** It was found calling `computeNetwork(state.tiles, el, state.center, state.nodes)` with the default `crossCenter: true` even though it only runs once `isPathComplete` (which itself correctly uses `false`) has already confirmed `el`'s path is complete — the mismatch meant the LIT set (used purely for the visual glow, not the win check itself) could still tunnel through the Orrery into a completely different, unconnected, possibly-impure fragment and light it up as if it were part of a finished path. `litKeys` now mirrors `isPathComplete`'s own traversal exactly (`crossCenter: false`) so "lit" and "complete" always agree on which tiles constitute a path — a lit/glowing tile always genuinely sits on a connected run from that element's own node to the Orrery, never a disconnected fragment that merely happens to also touch the center hub.
- **`GameState` carries a family of transient "seq" event counters** (`shootingStarSeq`/`lastShootingStarEvent`, `eclipseEventSeq`/`lastEclipseEvent`, `asteroidEventSeq`/`lastAsteroidDestroyedTiles`, `discardEventSeq`, `starDeckShuffleSeq`, `eclipseDeckShuffleSeq`, `shieldBlockSeq`/`lastShieldBlock`, `chainEventSeq`/`lastChainEvent`, `surgeEventSeq`/`lastSurgeEvent`) that exist purely so `GameScreen` can react to "this specific kind of thing just happened" without parsing `s.log` strings. Each is bumped by the engine code that causes the event and never decremented; `GameScreen` diffs them against a ref-held baseline from the previous render (`eventBaselineRef`) to fire a flash or a `FlightLayer` ghost animation exactly once per real occurrence, and skips all of it on the very first post-mount diff (baseline starts `null`) so the initial hand deal / board generation doesn't spuriously animate. If you add a new "flash on X" or "fly a card on Y" effect, follow this same pattern rather than inferring the event from a state diff — diffing raw game state (e.g. hand length) is fragile because multiple different actions can produce the same-looking delta (see the Sagittarius/Cosmic-Draw note below). `surgeEventSeq`/`lastSurgeEvent` (`{ x, y, element }`) is the newest of these — bumped once in `reducer.ts`'s `PLACE` case whenever `card.element === p.element` (i.e. an Element Surge triggers at all, success or no-op branch alike), read by `GameScreen` to set a local `surgeTile` state for ~1s, threaded down through `GridBoard` to the one matching `TileView` as a `surging` boolean. `TileView` derives the flash color from `tile.card.element` at render time rather than trusting `lastSurgeEvent.element` directly — the card just landed there, so it's guaranteed present.
- **`GameState.messageLog` is a second, shorter, curated log alongside the full `GameState.log`**, populated via `important(s, msg)` (in `engine/log.ts`, alongside `log`/`logGroup`) — call it WITH the same message text right after (or instead of, for messages that only exist for this feed) the matching `log(s, msg)` call, never as a replacement for it. It's the source feed for genuinely notable events — Eclipse cards resolving, shooting stars, asteroid shifts (just the header, not every per-tile "↳" sub-line), chain-of-3+, path completions, corruption decay, closed loops, Element Surge, a Stasis reboot, win — while staying silent through the routine "moves to", "channels a card onto", "purifies", turn-announcement chatter that dominates the full Event Log. `damage()` in `eclipse.ts` only mirrors into `messageLog` for STANDALONE calls (no `out` array passed) — a grouped call (e.g. from the asteroid's per-tile sweep) already has its own header marked important by the caller, so mirroring every individual damage sub-line too would just be redundant clutter in the compact feed. Capped at 20 entries (vs. the full log's 40) since a single grouped event can otherwise fill it fast. If you add a new notable event, mirror its `log()` call with `important()` using the same judgment: is this a world event a player should notice at a glance, or is it routine per-turn narration? `important()` also increments an uncapped `GameState.messageSeq` counter every call — see the `StatusMessage` bullet below for why.
- **`StatusMessage` (next to `DeckTray`, above the board) shows exactly one notable event at a time, in larger legible text, rather than a scrollable log** — a deliberate redesign away from an earlier always-visible multi-line "Message Log" panel. `GameScreen`'s seq-diffing effect computes, on every `messageSeq` change, exactly how many of `messageLog`'s front entries are new (`state.messageSeq - prev.messageSeq`, since a single dispatch can call `important()` more than once — e.g. a placement that both crosses the chain-of-3 threshold and triggers an Element Surge) and hands that batch to `StatusMessage` via a changing `batchId` (a plain incrementing number, NOT the `messages` array's own referential identity — an array literal is a new reference every render regardless of content, which would otherwise re-trigger the enqueue effect on every unrelated re-render). `StatusMessage` queues incoming batches and displays one message at a time, each held for `DISPLAY_MS` (3.2s) before fading out and advancing to the next — so simultaneous events each get their own legible turn instead of only the last one surviving or all of them being unreadable at once. **The messageSeq diff is deliberately NOT gated behind the "skip the first post-mount render" guard that every other seq-diffed event in the same effect uses** — those other events (draw-flight, chain glow, etc.) would spuriously replay the initial hand deal/board setup if diffed against a null baseline, but the very first `important()` call of a fresh game (`initGame`'s "The Orrery awakens..." message) has no prior on-screen message for it to jump away from, so skipping it there would just mean the player never sees it. The messageSeq branch computes its own `prevMessageSeq = prev ? prev.messageSeq : 0` (falling back to 0, not skipping outright) specifically so it still fires on the very first render after mount.
- **`StatusMessage`'s fade-in timer must live in a SEPARATE `useEffect` from the one that pops the next message off the queue, even though both key off `current` becoming non-null.** The queue-popping effect's own dependency array includes `current` (to know when it's safe to advance) — if the same effect also started a `setTimeout` to fade the message in, that effect's OWN `setCurrent()` call changes `current` on the next render, which re-runs the effect (since `current` is one of its deps) and fires its cleanup FIRST, canceling the fade-in timer before its 20ms ever elapses. The message would then get "stuck" at `opacity: 0` forever with correct text underneath — a real bug hit once already, easy to reintroduce if these two concerns get merged back together for the sake of DRY-ness. Splitting them into `useEffect(..., [queue, current])` (pop) and `useEffect(..., [current])` (fade in/out/clear) means the second effect only re-fires when `current` genuinely changes to a new value, not as a side effect of its own sibling effect's state update within the same render pass.
- **Shooting stars are placed one per board quadrant (relative to the Orrery) — `SHOOTING_STAR_COUNT` (4) now exactly equals the 4 quadrants, so every quadrant always gets a star, unlike the earlier 3-star design which deliberately left one quadrant empty.** `placeShootingStars` (in `board.ts`) buckets candidate tiles into 4 quadrants via `quadrantOf(x, y, center)` (`(x < center.x ? 0 : 1) * 2 + (y < center.y ? 0 : 1)`), then picks one star per quadrant (in a shuffled quadrant order). Candidates are filtered up front by `SHOOTING_STAR_NODE_MARGIN` (a star may never spawn within that many tiles of ANY element node — a hard constraint baked into the candidate pool itself, never relaxed) in addition to the existing `SHOOTING_STAR_CENTER_MARGIN`. Per-quadrant selection ALSO prefers a candidate satisfying `SHOOTING_STAR_NODE_PATH_MIN`/`MAX` — for whichever ACTIVE element's node falls in that quadrant (`nodesByQuadrant`, also built from `quadrantOf`, but deliberately scoped to `activeElements` only, not every element — see below), the detour distance node → star → Orrery (Manhattan) must land in that range — combined with `SHOOTING_STAR_MIN_SPACING`, but this pair of constraints is soft: `placeShootingStars` tries path-length+spacing, then spacing alone, then path-length alone, then any candidate in the quadrant, before finally falling back to ignoring the quadrant assignment entirely (still respecting node/center margin) only if a quadrant genuinely has no eligible tile — this can never fail to place all `SHOOTING_STAR_COUNT` stars, it just relaxes the softer constraints on an unlucky board layout (the node-path-length constraint specifically holds ~85-90% of the time across random boards in practice — see `board.test.ts`'s statistical test). **`nodesByQuadrant` here is built from `activeElements`, NOT every element, precisely so the rule is always scoped to a single active player's own route — since `pickCenterAndNodes` (see its own bullet above) now guarantees every active player's node lands in a distinct quadrant, a quadrant can hold AT MOST ONE active element's node, so the path-length rule can never be "satisfied" by two different players sharing the same shooting star.** An INACTIVE element's node can still share that same quadrant (nothing constrains it), but since it's excluded from `nodesByQuadrant` here, its distance to the star is simply never checked — it doesn't tighten or loosen the active player's own rule at all.
- **In a 2-3 player game, `assignShootingStarPowerUps` (in `board.ts`, exported for direct unit testing) preferentially routes the power-ups in `SHOOTING_STAR_POWER_UP_PRIORITY` (constants.ts; default `[BONUS_AP, TRACKER_DOWN, BONUS_HAND]`, reorder freely) into whichever quadrants contain an ACTIVE player's own element node** — so a Guardian is more likely to find a beneficial power-up in their own corner of the board instead of one being wasted on a quadrant nobody's element is anywhere near. It walks the priority list in order, assigning one type per distinct active quadrant (shuffled so it's not always the same physical star that gets first pick); if two active players' nodes share one quadrant, only that many priority items get guaranteed placement (e.g. just the first) and the rest fall into the ordinary leftover shuffle along with every non-active quadrant's star. A 4-player game always uses every element (no "off" quadrant to avoid the treatment), so `activeElements.length >= ELEMENTS.length` short-circuits straight to a full shuffle of all 4 types. `initGame` computes `activeElements` from the deduplicated set of `setup`'s chosen signs' elements (never actually duplicated within one game — see `SetupScreen`'s own guard) and threads it through `makeBoard(rng, activeElements)`; `makeBoard`'s own default (`activeElements = ELEMENTS`) exists purely so ad-hoc callers (tests) that don't care about player-count-aware prioritization don't need to pass one.
- **The 4th shooting star power-up, HEAL_UNLOCK, is a permanent global unlock (`GameState.selfHealUnlocked`, same "never cleared, just set true once" shape as `apBonus`/`handSizeBonus`) rather than an instant effect.** Once triggered, EVERY Guardian (not just whoever triggered it) may restore `SHOOTING_STAR_SELF_HEAL_AMOUNT` (1) HP by ending a turn in which they took no other action while below full HP. This is gated by `GameState.actedThisTurn`, a per-turn flag reset to `false` the moment a new player becomes active (at the very end of `END_TURN`, alongside `ariesUsed`/`libraUsed`/etc.) and set to `true` unconditionally for every OTHER action type right after `gameReducer`'s clone step (`if (action.type !== "END_TURN") s.actedThisTurn = true;`) — the exact same "safe because an invalid action returns the original, pre-clone `state`, discarding this mutated `s` entirely" pattern `lastActedPlayerId` already uses (see that bullet elsewhere in this file), so a rejected/no-op action dispatch never wrongly counts as "taking an action." The actual heal check runs in `END_TURN`, keyed to `p` (the player ending THIS turn, captured before the active-player reassignment) and BEFORE `s.actedThisTurn` gets reset for whoever's about to become active — same "key it to the turn that's actually ending" ordering as the corruption-decay sweep directly above it in the reducer, and for the same reason: checking `newActive` instead would silently check the WRONG player's flag. Drives a `selfHealSeq`/`lastSelfHealEvent: { playerId }` seq-event pair (same diffing pattern as `shieldBlockSeq`/`lastShieldBlock`) that `GameScreen` turns into a ~1.2s `selfHealFlashPlayerId`, threaded down to `ControlPanel` and applied as a `caHeartRestore` (styles.ts) pulse-and-glow animation directly on the `♥` hearts — both on the always-rendered roster row (the common case, since the healed player has always just handed control to someone else by render time) and on the top active-player status card's hearts (only visually relevant in the rare case where the SAME player remains active across turns because every other Guardian is in Stasis).
- **Hand-length diffing for the draw-flight animation only fires for the *active* player.** `GameScreen` tracks each player's hand length across renders and only animates a draw (Star Deck icon → hand panel) when the *currently active* player's hand grew relative to their own last-seen count — an `END_TURN` refill happens for the player who just finished their turn, by which point `state.active` has already moved on and there's no visible hand panel to animate into anyway, so that case is a deliberate no-op, not a bug. Cosmic Draw's (renamed from Cosmic Discard) same-turn redraw is the main case that actually animates.
- **Asteroids now respect three more constraints** (`triggerAsteroidShift` in `reducer.ts`): they can't choose a destination tile occupied by a player, and — via `computeLunarShieldTiles` (shared with `damage()`'s Cancer-shield check) — can't destroy cards on tiles within a Cancer Guardian's Lunar Shield radius, nor on Shooting Star tiles; both are folded into the same tile-skip branch that already spares completed-path tiles, each with its own log line. Cards actually destroyed by the sweep are now pushed into `s.starDiscard` (previously they just vanished) so they can be reshuffled back into the Star Deck later, and are recorded in `lastAsteroidDestroyedTiles` for the discard-pile flight animation.
- **Sagittarius's Astral Arrow has no fixed range anymore.** `validPlacement` special-cases `SAGITTARIUS` entirely: instead of the normal `manhattan(player.position, target) > 1` gate, it computes `reachableFrom(tiles, player.position)` (a connector-based BFS treating any node/the Orrery as an always-open hub — this is "the connected component containing wherever the player is currently standing," which board.ts also uses for `tracePathBetween`) and requires the target be adjacent to *some* tile in that set, not just the player's own position. `placementCost` was simplified to a plain `manhattan > 1 ? 2 : 1` for everyone — this is behaviorally identical to before for every other sign (they can never legally reach distance >1 anyway) but now also correctly prices Sagittarius's remote placements.
- **Every sign — not just Sagittarius — can build onto a fragment that's been severed from every node/the Orrery** (e.g. an asteroid cut a path in two, stranding a player on the disconnected piece). `validPlacement` always computes `reachable = reachableFrom(tiles, player.position)` (previously this was Sagittarius-only, named `sagittariusReachable`) and the connector check inside `check()` accepts a neighbor via `union.has(...) || reachable.has(...)`, not just `union` (which is only built from node-/center-rooted networks and would never contain a severed fragment at all). Since a player's own `reachable` set is always a subset of `union` on a normally-connected board, this purely adds coverage for the severed-fragment case and never removes a previously-valid placement. `reachableFrom`/`tracePathBetween` also both special-case a bare (cardless) starting tile as "open in every direction" — without it, a player standing on a broken, cardless spot (their card destroyed by an asteroid, say) couldn't expand outward at all, since the BFS's per-direction gate normally requires the *current* tile's own card to have a connector facing that way. `getValidMoves` (in `rules.ts`) uses this same isHub/bare-tile-open logic per hop, so movement always requires the CURRENT tile's own connectors to actually face the next step — a physically-touching but unconnected neighbor (mismatched connector shapes) is never a valid move, no matter how many hops away.
- **Move covers multiple tiles in a single click, not just one, when the player has enough AP.** `getValidMoves` returns `Map<string, MoveReach>` (`{ cost, parent }`) instead of a plain `Set<string>` — a BFS from the player's own tile using the identical per-hop connector-matching rule described above, generalized to keep expanding for as many hops as `s.ap` allows (1 AP per tile stepped through, same rate a single Move always cost), tracking both the cumulative cost to reach each tile and which tile precedes it on that path. `hasAnyAction`'s `getValidMoves(s).size > 0` check and every existing `.has(key)` test still work unchanged since `Map` supports both. The reducer's `MOVE` case looks up the clicked destination's `MoveReach`, deducts its `cost` (not a flat 1), and walks the `parent` chain back to the player's starting tile marking **every intermediate tile visited** (`p.visited`) — a compressed multi-tile move is mechanically still several individual moves, and Purify eligibility depends on having personally walked onto a tile, not just landed on the final one. The log line only mentions a tile count when `cost > 1` (`"moves 3 tiles to (x,y)."`), so a plain adjacent move still reads exactly as it did before. Board tiles reachable at `cost > 1` get the same small AP-cost badge Sagittarius's remote placements use (see below) — `TileView`'s prop was generalized from a Sagittarius-only boolean (`apCost2`) to `apCostBadge?: { cost: number; tooltip: string }`, computed once in `GameScreen` per mode (place vs. move never overlap, so one `Map` covers both cases).
- **Cancer's Lunar Shield is now visible on the board**, not just a damage-check side effect. `computeLunarShieldTiles(s)` (in `eclipse.ts`, shared by `triggerAsteroidShift` and `GameScreen`) returns every tile on or adjacent to a non-Stasis Cancer Guardian; `GridBoard`/`TileView` render a soft blue inset ring + a small 🌊 badge on those tiles. This is a derived, recomputed-every-render set, not stored on `Tile` — don't confuse it with `Tile.isShielded` (Virgo) or try to persist it.
- **The Star/Eclipse/Discard pile icons (`DeckTray`) are the anchor points for `FlightLayer`'s ghost animations**, not just a readout. `GameScreen` holds a `React.RefObject` for each pile plus the (currently visible) hand panel and the board wrapper, and computes flight start/end points via `getBoundingClientRect()` at the moment an event fires — there's no persistent layout math, so if you resize/reposition `DeckTray` the flights automatically still land in the right place. Tile-level flights (Eclipse card reveal, asteroid-destroyed cards) look up their tile's DOM node via `document.querySelector('[data-tile="x,y"]')`, which `TileView` sets on its root element specifically for this purpose.
- **Purify's board highlights are filtered by affordability, not just validity.** `getValidPurifyTargets` (path-based, per the bullet above) answers "is this tile a legal purify target at all"; `getAffordablePurifyTargets` (also in `rules.ts`) narrows that further to tiles whose `purifyCost` the player can actually pay right now. `purifyCost` is a flat 1 AP everywhere (0 for Taurus's free first use each turn) — it used to charge 2 AP for a tile the player wasn't currently standing on, but that distinction was removed so Purify costs the same regardless of location; the affordability filter itself still matters at exactly 0 AP. `GameScreen`'s `highlights` memo uses the affordable variant for purify-mode board highlighting; `purifyDisabledReason`'s messaging intentionally still uses the unfiltered target set, since "you have a target but can't afford it" is exactly the case it's explaining.
- **Purify mode stays armed after a successful purify if more affordable corrupted tiles remain**, instead of dropping back to no-mode after every single tile — lets a player clean up several corrupted cards in a row without re-clicking the Purify button in between each one. `GameScreen`'s `onTileClick` purify branch calls `gameReducer(state, { type: "PURIFY", x, y })` as a **pure preview** (its return value is only used to check `canPurify(next)`; the actual state-updating `dispatch({ type: "PURIFY", x, y })` happens as its own separate call right after, exactly as it always did) and only calls `resetUi()` when `canPurify(next)` comes back false. This is safe specifically because `gameReducer` is a pure function — calling it here mutates nothing and doesn't touch React state, so there's no risk of double-applying the purify. The same technique (a throwaway `gameReducer` preview call purely to inspect what WOULD happen) is a reasonable pattern to reach for anywhere else a UI decision needs to know the post-action state before the real dispatch's result is available on the next render.
- **AP-cost action buttons show a small rotated-diamond badge with the cost inside** (`ApBadge`, mirroring `ApPips`' same rotate-45 motif), via `NeonButton`'s optional `apCost` prop — a plain number for fixed-cost actions (Move, Bastion Ward, Terraform Hand), a live-updating number for cost-varying ones (Cosmic Draw's `discardCost`, which drops to 0 for Libra's free first use), or a string like `"1-2"` for Purify, whose actual cost depends on which tile gets picked. Free actions (Discard to Heal, Rotate, End Turn) simply omit `apCost` and get no badge. Don't fold the cost back into the button's text label (`"Move · 1 AP"`) — that was the pre-badge convention and is exactly what this replaced. The same `ApBadge` also appears directly ON THE BOARD, for any highlighted tile costing more than 1 AP: Sagittarius's remote (2 AP) placements, and multi-tile Move destinations more than 1 step away (see that bullet above). `GameScreen`'s `apCostTiles` memo builds one `Map<string, { cost, tooltip }>` covering whichever of those two cases applies to the current mode (they never overlap), threaded through `GridBoard` to `TileView`'s `apCostBadge` prop, which renders a small top-centered `ApBadge` with a per-tile tooltip explaining the cost — 1 AP tiles get no badge, matching the "cost is implicit unless it varies" convention everywhere else.
- **Star Cards that close a loop in the connector graph (any shape that encloses an area, a rectangle being the simplest) become permanently immune to Eclipse Corruption and the traveling asteroid.** `findEnclosedTiles(tiles)` (in `board.ts`) builds the undirected graph of card-to-card connector edges (any element, node/center hubs deliberately excluded as vertices — this is specifically about Star Cards forming a shape, not routing through hubs) and runs Tarjan's bridge-finding over it: an edge that ISN'T a bridge lies on some cycle, and the union of such edges' endpoints is the "enclosed" set. `reducer.ts`'s `PLACE` case calls this after every placement and sets `Tile.isEnclosed = true` for anything newly found — it's a one-way flag, never cleared, so breaking the loop later (e.g. an asteroid destroying a different card that used to close it) doesn't strip immunity already earned. `eclipse.ts`'s Corruption target filter and `triggerAsteroidShift`'s destination-candidate/per-tile-sweep checks both skip `isEnclosed` tiles the same way they already skip `isLocked`/`isShielded`/completed-path tiles. Rendered as a soft amber inset ring + a small 🔒 badge in `TileView`, reading directly off `Tile.isEnclosed` (unlike Lunar Shield, this one IS stored on the tile, not recomputed per render).
- **A tile that's ever been Purified becomes permanently immune to Corruption**, via `Tile.isPurified` — set in `reducer.ts`'s `PURIFY` case both for the directly-targeted tile and for every tile Leo's Solar Flare sweeps along the way. `eclipse.ts`'s Corruption target filter skips `isPurified` tiles the same way it already skips `isLocked`/`isShielded`/`isEnclosed`/completed-path tiles. Unlike `isEnclosed`, this flag IS cleared — specifically when the traveling asteroid destroys that tile's card (in the same `t.card = null` reset block in `triggerAsteroidShift`) — since the immunity was earned by purifying THIS card, and a brand-new card placed there later hasn't been purified itself. Rendered in `TileView` as the card glyph rendering a touch lighter (`lerpColor` toward white) plus a soft pale inset ring (skipped when `isEnclosed` is also true, since that ring already implies the same immunity and takes visual priority).
- **`fmtNum` (in `engine/log.ts`) must wrap every log-message interpolation of a value that could carry floating-point residue** — `s.tracker` and any Eclipse Card amount scaled by `ECLIPSE_EFFECT_SCALE_4P`/`ECLIPSE_SURGE_CORRUPTION_SCALING` are deliberately left as raw, non-integer floats in `GameState` (see the 4-player-balance bullet below — this is intentional, not a bug, and tests assert against the unrounded value via `toBeCloseTo`). Printing one of these directly with a bare `${...}` template interpolation reproduces JS's raw float representation (`2.0999999999999996`) instead of a clean `2.1`. `fmtNum` only affects the STRING shown in `s.log`, never the underlying state value — don't reach for `Math.round()` on the state itself, that was tried once and broke a test that specifically asserts the fractional value survives (`3 * 0.7` must stay `2.1`, not round to `2`).
- **Edge nodes are re-rolled (bounded, up to 200 attempts) until every pair is at least `MIN_NODE_SEPARATION` (4) tiles apart AND every ACTIVE player's node lands in a distinct board quadrant.** `pickCenterAndNodes(rng, activeElements)` in board.ts wraps `pickCenterAndNodesOnce()` in a retry loop checked by BOTH `nodesWellSeparated()` and `nodesInDistinctQuadrants()`. AIR/EARTH (opposite rows) and WATER/FIRE (opposite columns) are always far apart structurally, but adjacent-edge pairs — e.g. AIR/WATER, which share the (0,0) corner — could otherwise land right on top of each other (each edge's position is picked independently), just uncomfortably close, or (the newer concern) on merely a different EDGE while still landing in the SAME quadrant relative to the Orrery (`quadrantOf`, shared with the shooting-star placement logic below). `nodesInDistinctQuadrants` only checks `activeElements` — the elements actually in play this game — not every element; an INACTIVE element's node is free to share a quadrant with anyone, since no Guardian ever spawns there, so a 2-3 player game's board generation is less constrained (and thus needs fewer retries) than a 4-player one. Separately, `SetupScreen` already disables starting a game with two players sharing an element (`duplicateElements` check), so two players can never spawn on the exact same node tile regardless of this board-generation constraint. `makeBoard`'s own default (`activeElements = ELEMENTS`) means an ad-hoc caller that doesn't pass one (tests, mainly) gets the strictest version — all 4 nodes guaranteed in 4 distinct quadrants — which is a reasonable default and also why `board.test.ts`'s shooting-star node-path-length statistical test now always finds exactly one node per quadrant rather than sometimes finding two.
- **Eclipse Void's target filter must exclude Shooting Star tiles explicitly** (`!t.isShootingStar` in `eclipse.ts`'s VOID branch) — a Shooting Star tile has no card, so it otherwise looks exactly like a legitimate empty tile to Void's `!t.card` check, and a black hole spawning there would silently destroy the power-up underneath it.
- **Placement highlights are filtered by affordability the same way Purify's are.** `getAffordablePlacements` (in `rules.ts`) narrows `getValidPlacements` down to tiles whose `placementCost` the player can currently pay, mirroring `getAffordablePurifyTargets`. This only actually changes anything for Sagittarius in practice — every other sign is range-capped to `manhattan <= 1`, which always costs 1 AP, but Sagittarius can target a 2-AP tile that shouldn't be highlighted as selectable once they're down to 1 AP. `GameScreen`'s `highlights` memo uses the affordable variant for place-mode board highlighting.
- **Hand cards with nowhere affordable to place them are grayed out and unselectable, not just silently useless once armed.** `GameScreen` computes `unaffordableCardIndices` (a `useMemo` over `active.hand`, calling `getAffordablePlacements(state, i).size === 0` per index — the SAME per-card affordability check the board highlighting above uses, just checked without a card already armed) and passes it to `CardHand` as `unaffordableIndices`. `CardHand` only actually applies the grayed/`disabled` styling when tapping the card would try to ARM it for placement (`mode !== "discard" && mode !== "scorpioHeal"` — in those two modes, tapping a card means something else entirely that isn't gated on placement affordability at all: marking it for Cosmic Draw, or spending it on a Scorpio heal). At 0 AP every card is unaffordable (since every placement costs at least 1 AP) and the whole hand grays out at once; for Sagittarius specifically, a card whose only valid targets are all 2-AP-away tiles can go gray even with 1 AP still banked, while a different hand card with a 1-AP-adjacent target stays selectable — this is why the check is per-card, not just a single "if AP > 0" gate.
- **4-player games get two specific balance breaks** to offset needing 4 separate completed paths to win (a meaningfully harder race against the Eclipse Tracker than 2-3 player games): completing any path reduces the tracker by `PATH_COMPLETE_TRACKER_REDUCTION_4P` (10%, logged inline with the "path blazes into the Orrery" message — `reducer.ts`'s `PLACE` case, gated on `s.players.length === 4`), and every Eclipse card's tracker-affecting amount (Corruption/Void's "no valid target" bump, Surge's total amount including corruption scaling) is multiplied by `ECLIPSE_EFFECT_SCALE_4P` (0.7, i.e. a 30% reduction) via `eclipseEffectScale(s)` in `eclipse.ts`. Both are no-ops (multiplier of 1, no bonus reduction) for 2-3 player games. These produce non-integer tracker deltas by design (e.g. a 3% bump becomes 2.1%) — `s.tracker` isn't integer-constrained anywhere else either.
- **Board generation is seedable for replaying a specific starting layout.** `GameState.seed` holds whichever string was used (user-typed via `SetupScreen`'s "Board Seed" field, or an auto-generated `randomSeedString()` if left blank) — `initGame(setup, seed?)` hashes it (`hashStringToSeed`, xmur3) into a 32-bit int and seeds a `mulberry32` PRNG (`utils/rng.ts`), then threads that single `rng` function through `makeBoard(rng, activeElements)` (Orrery/node positions, asteroids, shooting stars) and `buildStarDeck(rng)`/`buildEclipseDeck(rng, activeElements)` (initial deck shuffle/rotations). Every other `Math.random()` call in the engine (deck reshuffle-on-empty, Eclipse targeting, asteroid shift targeting) is deliberately left alone — a seed reproduces the *starting board*, not a bit-for-bit replay of an entire playthrough, since that would also require identical player actions throughout. `shuffle`/`buildStarDeck`/`buildEclipseDeck`/`makeBoard` all default their `rng` param to `Math.random`, and `makeBoard`/`buildEclipseDeck` both default `activeElements` to `ELEMENTS` (all 4), so ad-hoc callers (tests, mainly) don't need to pass either. The seed is displayed in `ControlPanel` next to the turn counter with a copy-to-clipboard button (`SeedDisplay`).
- **Eclipse Damage Cards (`EclipseCard.type === "DAMAGE"`) hit every living Guardian of specific element(s) directly, by design — this is the one Eclipse card type that does NOT pick a single random tile/target the way Corruption/Void do.** The card templates live in `DAMAGE_CARDS` (constants.ts, difficulty-tuning block, freely editable — `{ message, elements, hpLost }`) rather than being hardcoded in the engine; `buildEclipseDeck(rng, activeElements)` samples `ECLIPSE_DAMAGE_CARD_COUNT` (4) of them **with replacement** (not shuffle+slice) since the count can exceed the eligible pool size in a 2-3 player game, and copies each sampled template's fields onto a fresh `EclipseCard` (`damageElements`/`damageMessage`/`damageHpLost`) rather than referencing the template by id. Before sampling, templates whose `elements` don't overlap `activeElements` at all are filtered out entirely — a card that can never find a victim in THIS game is never even a possible draw, not just a wasted one. `resolveEclipse`'s DAMAGE branch (`eclipse.ts`) filters `s.players` for non-Stasis matches and calls the existing `damage()` helper on each (so Cancer's Lunar Shield/Stasis-transition/HP-clamp all apply identically to any other damage source) — each victim's own damage line is collected via `damage()`'s `out` param and flushed together with `logGroup` so they read top-to-bottom under the header instead of reversed (same pattern as the asteroid sweep). The card's own `damageMessage` is flavor text ONLY — the `"🌑 Eclipse:"` prefix every log line gets is applied once at the log call site (`` `\u{1F312} Eclipse: ${c.damageMessage}` ``), not baked into the stored message, so every Eclipse card type's log lines share one consistent prefix convention regardless of type. `lastEclipseEvent.kind` gained a `"DAMAGE"` value (always with `x: null, y: null`, since there's no single tile to point at) — `GameScreen`'s flight animation renders it with a `💥` glyph and, having no tile target, flies toward the Orrery's own tile (see the next bullet).
- **The Eclipse-card flight animation (deck icon → target) must fly toward the Orrery's OWN tile position when there's no specific target tile, not the board wrapper's geometric center — this was a real bug.** `GameScreen`'s seq-diffing effect used `boardWrapRef.current?.getBoundingClientRect()` as the fallback destination for Surge/Damage cards and Corruption/Void's "no valid target" case (`lastEclipseEvent.x/y === null`) — but the Orrery is randomly placed within its center zone, not necessarily anywhere near the visual midpoint of the whole 19x11 grid, so the flight would frequently land somewhere that isn't the Orrery at all. The fix looks up `` document.querySelector(`[data-tile="${state.center.x},${state.center.y}"]`) `` first and only falls back to `boardWrapRef` if that tile element genuinely can't be found (shouldn't normally happen).

## Testing

`npm test` (or `npm run test:watch`) runs Vitest against `src/**/*.test.ts` (config in `vitest.config.ts`) — this includes `src/utils/rng.test.ts` and `src/i18n/index.test.ts` alongside everything under `src/engine/`. Tests are Node-environment only, no DOM — everything covered is pure functions over plain data, so there's no need for jsdom here (this includes `i18n/index.ts`'s plain `t()`/`setLocale()`/`getLocale()`, since they don't touch React). Pinned to **Vitest 1.x deliberately**: Vitest 4's bundled `rolldown-vite` requires Node ≥20.19 (`node:util`'s `styleText` export), and this environment runs Node 18.16 — if you ever bump vitest, confirm the Node version moved too, or you'll get a cryptic `SyntaxError` on startup rather than a version-mismatch error.

- **A test asserting a low-probability random event (e.g. the traveling asteroid sweeping one specific tile out of ~200) needs a retry loop across many fresh games, not a single attempt.** A single roll can plausibly miss ~15% of the time even across 40 attempts if the true per-attempt hit chance is only a few percent — `reducer.test.ts`'s `isPurified`-reset-on-asteroid-destruction test retries up to 500 fresh games and only asserts once it actually observes the event, which pushes the false-failure probability low enough (~1e-7) to be a non-issue in practice. This is different from the OTHER "tolerant" pattern already documented below (asserting nothing and returning early on a miss) — retrying is strictly better whenever the assertion is the actual point of the test, since a silent early-return can let a test go years without ever really exercising its own claim.

Gotchas hit repeatedly while writing these, worth knowing before adding more:
- **A straight line from a node rarely reaches the Orrery.** Nodes are equidistant from the center by Manhattan distance, not row/column-aligned with it — walking a node's own `.dir` direction in a straight line almost never lands on the actual center tile. Use `board.test.ts`'s `buildPathToCenter` (an L-shaped walk: align one axis, then the other, stopping one tile short of center) for any test that needs `isPathComplete` to actually be true, not the simpler straight-line `buildChainFromNode`.
- **Row/column 3-7 (the Orrery's possible center zone) and rows 0/10 (AIR/EARTH's node rows) are unsafe to hand-pick coordinates in.** A test that relocates nodes or hand-builds geometry (e.g. `reducer.test.ts`'s wildcard-convergence test, via its `relocateNode` helper) should pick a row outside both ranges (row 1, 2, 8, or 9) so it can never coincidentally collide with the real Orrery or another element's node on that random board — a collision doesn't crash, it just quietly changes what the test is actually exercising (e.g. the hand-built chain accidentally completing the whole game), which is far more confusing to debug than a bounds error. `NodeInfo.dir` itself is safe to leave stale after relocating — it only matters for `makeBoard`'s forbidden-zone math, never for connectivity/purity.
- **A hand-picked perpendicular "branch" offset needs a bounds check.** `DIRS[node.dir]` tells you the chain's own axis, but the perpendicular test direction (`+1` on the other axis) can walk off the board if the node happens to sit right at that edge — see `board.test.ts`'s `safePerp`, which falls back to `-1` when `+1` would go out of bounds.
- **`PLACE` without an interleaved `MOVE` only works for the first card.** Every sign except Sagittarius requires the pawn within 1 tile of the target, so a test loop building a multi-card chain needs `MOVE` after each `PLACE` (Sagittarius is the one legitimate exception — see its own test using bare repeated `PLACE`s from a stationary pawn).
- **A Fire-element player placing a Fire card auto-cleanses one adjacent corrupted tile** (Element Surge, `SURGE_META.FIRE`) — if a test corrupts a tile and then has an Aries/Leo/Sagittarius player place a Fire card next to it, the corruption may vanish before the assertion you're actually testing for gets a chance to matter. Use a player whose own element differs from the card element being placed (e.g. Cancer placing Fire cards) when a test specifically needs corruption to *stick around*.
- **Log-order assertions should search the whole log, not just the top few lines.** `log()` unshifts (newest first), so an event that also happens to complete the game (win-check messages appended afterward) pushes everything else down further than you might expect — assert `s.log.some(...)`, not `s.log.slice(0, N).some(...)`, unless you've deliberately ruled out any possibility of a cascading side effect.

## Known CSS/layout gotcha: sizing a fixed-aspect-ratio box inside a flex/grid cell

`GridBoard`'s WIDTH×HEIGHT tile grid needs to be "the largest WIDTH:HEIGHT-ratio box that fits in whatever space is left." The naive approach — `w-auto h-auto aspect-[19/11]` plus a `max-h` cap, inside a `flex items-center justify-center` or CSS-grid `place-items:center` parent — **looks correct (produces the right ratio) but silently shrink-wraps to the tiles' own min-content size instead of growing to fill the available box**, because `align-items`/`place-items` values other than `stretch` resolve auto-sized dimensions via fit-content, not via the parent's available space. This is easy to miss because the result still has the right proportions, just wrong (too small), and doesn't visibly break anything.

The fix in this repo is `hooks/useFitSize.ts`: it measures the parent element's `clientWidth`/`clientHeight` via `ResizeObserver` and computes the largest `{width, height}` at the given aspect ratio that fits, capped at a max, as explicit pixel sizes — avoiding CSS auto-sizing ambiguity entirely. If you touch the board layout, keep using this hook (or something equivalent) rather than reintroducing `aspect-ratio` + auto-sizing — verify by actually reading `getBoundingClientRect()` in a browser, not just by eyeballing a screenshot, since a too-small-but-correctly-proportioned board doesn't look obviously wrong at a glance.

**The same "don't trust a formula, measure the real DOM" lesson applies to `GridBoard`'s edge node labels (AIR/EARTH/WATER/FIRE text outside the board).** They used to be positioned via `(nodeIndex / (WIDTH-1)) * 100` as a `left`/`top` percentage of the OUTER wrapper (`ref`, sized by `useFitSize`) — but the actual node tile's center does NOT fall at that percentage of the wrapper's box, because the inner grid has its own padding (`p-1.5 sm:p-2.5`), border, and inter-tile gaps (`gap-0.5 sm:gap-1`), none of which the index-based formula accounts for, and all of which are responsive (different at the `sm:` breakpoint) — so the labels appeared slightly, consistently offset from their node. The fix (in `GridBoard.tsx`) is the same pattern as `useFitSize`: a `useLayoutEffect` + `ResizeObserver` that measures the ACTUAL rendered node tile's `getBoundingClientRect()` (found via its `data-tile="${x},${y}"` attribute, same lookup mechanism `FlightLayer`'s animations already use) relative to the outer wrapper's own rect, and stores the resulting percentages in state (`edgePercents`) rather than computing them from the node's index. This is correct regardless of how the grid's padding/gap/border CSS changes in the future, since it never has to be told about them — verify any edge-label change with `preview_inspect`/`getBoundingClientRect()` measurements, not just a screenshot, for the same "slightly wrong doesn't look obviously wrong" reason as the sizing gotcha above.

## UI conventions

- Keyboard shortcuts (M/P/V/H/T/C/E/B/Escape, plus 1-4 for hand cards) are wired in `GameScreen.tsx`'s single `useEffect`, guarded by the same availability checks the corresponding buttons use (`canPurify`, `canUseVirgoShield`, `canScorpioHeal`, `canConvertHandEarth`, AP costs). If you add a new action button, add its shortcut in the same place and keep the guard logic in sync with the button's `disabled` condition, or the shortcut will silently no-op or (worse) dispatch on an unavailable action. End Turn is `E`, not `Enter` — Enter (and Space) are reserved for the board tile cursor below, and reusing them for End Turn would make it impossible to confirm a keyboard-selected tile without also risking an accidental end-of-turn.
- **The entire game is keyboard-playable via a board tile cursor**, not just mode toggles. `GameScreen` holds `boardCursor: {x,y} | null`; Arrow keys move it one tile at a time (clamped to the board's bounds), snapping to the active player's own position on first press each turn. It's cleared to `null` (no visible outline) in three cases: whenever `state.active` changes (a new turn always starts fresh rather than inheriting the previous player's cursor), whenever `state.phase` leaves `"playing"` (the outline has nothing left to select on a win/loss screen and looks out of place lingering there), and on `Escape` (alongside the rest of `resetUi()`) — all three are handled by the same `useEffect([state.active, state.phase])` plus an explicit `setBoardCursor(null)` in the `"escape"` key-handler branch. Enter or Space then calls the same `onTileClick` used by a mouse click on that tile — safe to call unconditionally since `onTileClick` already no-ops on a non-highlighted tile. Rendered in `TileView` as a `cursorFocused` white dashed outline (`caCursorPulse`), independent of the cyan dashed "highlighted"/gold solid "previewed" styling so it stays visible regardless of mode. While `EndTurnConfirmModal` is open, the same `useEffect` short-circuits to just `Enter` (confirm) / `Escape` (cancel) and swallows every other key, so board navigation can't leak through underneath it.
- **The keyboard `useEffect`'s dependency array must include every piece of local UI state that `onTileClick`/the handler body reads, not just the ones that obviously change per keypress.** This bit us once already: `shieldPreview` was left out, so after the first Enter press armed a Virgo shield preview (via `setShieldPreview`, which doesn't touch `state`/`mode`/`selectedCard`/etc.), the effect never re-subscribed and the `onKeyDown` closure kept referencing the stale `shieldPreview === null` — the second Enter press re-armed the same preview instead of confirming it, since `onTileClick`'s `shieldPreview && shieldPreview.x === x && shieldPreview.y === y` check saw the old value. Mouse clicks never showed this bug because `GridBoard`'s `onClick` prop is a fresh closure every render. If a future action gains its own multi-step local state (mirroring Virgo's preview-then-confirm), add it to this effect's deps too.
- The hand renders **twice** in `GameScreen.tsx` — once in a `md:hidden` panel below the board (mobile, horizontal), once in a `hidden md:flex` vertical column next to the sidebar (desktop). Both use the same `CardHand` component; only one is visible at a time via Tailwind breakpoints, but both exist in the DOM. If you query the DOM in tests/automation, filter by `offsetParent !== null` (or similar) to avoid double-matching.
- `ControlPanel`'s End Turn button glows (`urgent` prop on `NeonButton`) when `hasAnyAction(state)` returns false — i.e., 0 AP, or no move/placement/purify/shield/heal/convert/discard is actually affordable. Cosmic DrawF being almost-always-available means this mostly triggers at exactly 0 AP; that's expected, not a bug.
- **Scorpio's heal is a two-step cross-component flow**: `GameScreen` puts the UI in `"scorpioHeal"` mode, the player taps a hand card (armed via `onHandSelect`, stored in `selectedCard`, *not* dispatched), then activates a roster row in `ControlPanel` (passed down as `healTargeting`/`onPlayerHeal`) to actually dispatch `SCORPIO_HEAL` with both the card index and target id. Roster rows are real `<button>` elements (Tab-focusable, Enter/Space-activatable, `disabled` when not a legal target) and do double duty — the same activation either targets a heal (when `healTargeting` is true) or toggles that player's expanded ability panel (`expandedId` local state) — so if you add a third behavior to roster-row activation, make sure the precedence between the two stays explicit. `CardHand`'s "armed" (popped-out + glowing) styling checks `mode === "place" || mode === "scorpioHeal"`, not just `"place"` — `onHandSelect` sets `selectedIndex` in both modes, and forgetting the second mode here was a real bug: the card became "selected" in state (the "tap a Guardian to heal" prompt showed correctly) while visually looking untouched.
- **The global board-cursor keyboard handler must defer to a focused `<button>` for Enter/Space**, or no button anywhere in the app (roster rows, hand cards, NeonButtons) can ever be activated by keyboard. `document.activeElement instanceof HTMLButtonElement` is checked first in the `"enter"`/`" "` case, returning immediately (no `preventDefault()`, no `onTileClick`) so the browser's native "Enter/Space clicks the focused button" behavior isn't suppressed. This was the second half of a real bug: our `window`-level `keydown` listener ran (and cancelled the event) regardless of what had focus, so Tab-ing to a roster row and pressing Enter/Space silently did nothing — the keypress was swallowed before the browser ever got to synthesize the button's click. Note synthetic (`dispatchEvent`-created) keyboard events never trigger this native click-synthesis regardless of `preventDefault` state, only trusted (real, OS-generated) ones do — so verifying this fix means checking `event.defaultPrevented` stayed `false` when a button has focus, not that a synthetic Enter actually produced a click.
- **A focused action button must give up focus, not just yield the keypress, when the player moves on to the board.** Building on the bullet above: `Escape` and every arrow-key press both call `(document.activeElement as HTMLButtonElement)?.blur?.()` (guarded by the same `instanceof HTMLButtonElement` check) before doing their normal job. Without this, a button could keep keyboard focus indefinitely after Tab-ing to it — the Enter/Space deferral means the board-cursor handler correctly leaves the button alone, but that also means Enter/Space would keep re-clicking that SAME button forever instead of ever reaching a board tile, since nothing else was moving focus away from it.
- **`TileView` layers a few purely-visual, non-interactive overlays on top of a tile independent of its `highlighted`/`previewed`/`chainGlow` border logic**: `activePlayerHere` (derived from `playersHere`/`activeId`, no new prop) draws a border in the active player's own element color around whichever tile they currently occupy, and `cursorFocused` draws the keyboard-cursor outline described above. Both are additive overlay `<div>`s (matching the existing `lunarShielded`/`isEnclosed` pattern) rather than mutations to the shared `borderColor`/`shadow` variables, specifically so they can't accidentally clobber (or be clobbered by) the interaction-affordance styling.
- **A tile's Star Card visually rises above any player token(s) standing on it while the tile is hovered — and so does its corrupted-tile Tooltip trigger, for the same reason.** `TileView`'s root element carries a `group/tile` class; the card's wrapper div only gets `group-hover/tile:z-20` when `playersHere.length > 0` (no-op otherwise, since there's nothing for the card to be obscured by) — `PlayerTokens` itself renders at a constant `z-10` outside the tile's clipped effects wrapper, so without this the token badge can partially cover the card glyph on tiles where a Guardian is standing. The corrupted-tile purple overlay's `Tooltip` wrapper needs the identical treatment (`className={"relative inline-flex" + (playersHere.length > 0 ? " group-hover/tile:z-20" : "")}`) for a different but related reason: it's not about the overlay being visually obscured (it renders fine either way) but about it being UNHOVERABLE — without the raised z-index, `PlayerTokens`' own hit-testing area sits on top and swallows the pointer, so the countdown tooltip could never actually open on a tile a Guardian was standing on. Hovering anywhere on the tile still triggers `:hover` on the `group/tile` root regardless of which child is topmost at that instant, which is what makes the CSS-only raise-on-hover trick work here rather than being a chicken-and-egg problem.
- **All hover/focus tooltips go through the shared `Tooltip` component (`components/Tooltip.tsx`), not the native `title=` attribute** — `title` is small, slow to appear, and unstyled, and (per the bullet above) it's also the ONE thing that survives synthetic/automated events, so don't reach for it as a testing workaround either. `Tooltip` always renders its wrapper `<span>` (even with no `text`, per its own doc comment) specifically because some callers — `PlayerTokens`, most notably — rely on that wrapper purely for layout/positioning (each stacked token's `left`/`bottom`/`zIndex` lives on the Tooltip wrapper), so conditionally skipping the wrapper when there's no tooltip text would silently break their position too, not just hide the popup. `className` on `Tooltip` REPLACES the wrapper's default `"relative inline-flex"` entirely rather than merging with it — any caller wrapping an element that needs its own absolute positioning (a tile corner badge, a header button pinned to an edge) must pass the FULL class string (including position) to `Tooltip`, not to `children`, or the child gets re-anchored to the new wrapper instead of the tile/page.
- **`Tooltip`'s popup is portaled straight to `document.body` and positioned with `position: fixed` from the trigger's live `getBoundingClientRect()`, not CSS-anchored inside the trigger's own DOM subtree.** This was a deliberate rewrite away from an earlier `absolute` + `group-hover`/`group-focus-within` version: several triggers live inside scrolling/clipped ancestors (`ControlPanel`'s `overflow-y-auto` sidebar, `TileView`'s clipped tile wrapper), and a CSS-anchored popup gets silently cut off by those ancestors' `overflow` the instant it extends past their box, no matter which `side` is picked — portaling escapes that class of bug entirely rather than requiring a `side` override per trigger. Position is computed in JS (`show()` reads `wrapperRef.current.getBoundingClientRect()` into state) on `mouseenter`/qualifying `focus`, so a scroll or resize while the popup is open would leave it pointing at a stale spot — handled by just dismissing it (`hide()`) on either, since these are momentary hover/focus popups, not persistent UI that needs to track position continuously. `side` still defaults to `"top"` and several known always-at-an-edge triggers (`GameScreen`'s Back button, the header `SeedDisplay`) still override it to point inward — that part of the contract is unchanged, since a `side` choice still affects which edge of the *viewport* a long tooltip might overflow, portal or not.
- **Tooltip visibility shows on real mouse hover and on keyboard focus, but deliberately NOT on a mouse click that merely happens to focus the trigger** — checked via `e.target.matches(":focus-visible")` inside the `onFocus` handler, relying on the browser's own heuristic that suppresses `:focus-visible` for pointer-triggered focus (but not Tab-triggered focus). Without this, clicking a button to activate it (e.g. a `ControlPanel` roster row) would leave its tooltip stuck open after the mouse moves away, since the button keeps DOM focus after being clicked and a naive `:focus-within`/focus-tracking check can't tell "focused because clicked" apart from "focused because tabbed to." This was a real reported bug, not a hypothetical.
- **`SetupScreen`'s "Zodiac Guardians Reference" panel is collapsible, defaulting to COLLAPSED** (`referenceExpanded` local state, `useState(false)`) — a plain toggle button (chevron rotates 90° via CSS `transition: transform`) shows/hides the same scrollable `max-h-40` list that used to always be visible. Defaulting to collapsed was a deliberate choice to declutter the setup screen (which already has a lot of vertical content — player count, seed, favorites, per-player cards) rather than trying to preserve the old always-visible behavior; there's no persistence (localStorage or otherwise) of the expanded/collapsed state across visits, unlike the setup choices in `utils/setupStorage.ts` — it's purely ephemeral per-mount UI state.

## Things that are NOT implemented (by design)

- No mid-game persistence — refreshing the page loses all game state (`useReducer` only). The one exception is `utils/setupStorage.ts`, which persists *setup screen* choices (names/signs per player count, last-used count, the last-typed Board Seed, and a nicknamed favorite-seeds list) to localStorage purely for quick replay — it never touches `GameState` itself, and all its reads/writes are wrapped to silently no-op if localStorage is unavailable (private browsing, etc.). `FavoriteSeed { id, seed, nickname }` entries are added/renamed/removed from `SetupScreen`'s own local state (`addFavoriteSeed`/`renameFavoriteSeed`/`removeFavoriteSeed`, each reading-modifying-writing the whole list); `addFavoriteSeed` is a no-op if that exact seed string is already saved, so there's no dedup step needed elsewhere. `loadLastSeed()`/`saveLastSeed()` prefill the Board Seed field with whatever was last typed (including blank, for "random") — saved once, in `start()`, alongside `saveLastSetup`, not on every keystroke.
- No AI/solo opponent; it's pass-and-play only.
- No animation/transition library — all motion is hand-written CSS `@keyframes` in `styles.ts`, plus `FlightLayer`'s manual two-phase (`start` → `flying` → `landed`) FLIP-style position transition for the deck/discard ghost animations.
